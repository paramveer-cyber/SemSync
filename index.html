01 High-Level Architecture

What This Application Is

SemSync is a semester-tracking web application for university students. Core features:

  • Course + evaluation management (grades, weightage, target grade computation)
  • Server-owned focus timer with full gamification (XP, streaks, achievements, daily goals)
  • Google Classroom integration for importing evaluations
  • Task center (localStorage-persisted, no backend)
  • Calendar view of upcoming evaluations
  • Achievement system with cinematic unlock animations
Key Architectural Facts
No Redis. No WebSockets. No SSR. The backend is a pure REST API. Real-time feel is achieved via polling — achievements every 45 s, timer sync every 10 s via HTTP heartbeat.

Design Philosophy

Scalable Monolith

Every domain (auth, courses, evals, focus) is a self-contained module with its own controller → service → queries chain. No micro-service boundary, but extraction later would be mechanical.

Server-Owned State

Timer elapsed time, XP, streaks, achievement awards — nothing trusted from the client. Server recomputes or anchors all values.

Events Table as Ledger

Gamification data derived by counting rows in the events table. History queryable, achievements re-evaluable, idempotency trivially implemented.

Cache-First Frontend

All course data loaded in one burst on first cache miss. Subsequent pages are instant (zero network). Invalidated explicitly after mutations.

Token Lifecycle (Frontend)

App boot
  │
  └─ AuthContext mounts
      POST /auth/refresh (httpOnly cookie)  →  access token stored in tokenStore.js (memory only)
      GET /auth/me                          →  user object in React state

API call (api.js)
  │
  ├─ 200 OK  →  return data
  │
  └─ 401 TOKEN_EXPIRED  (and not _retry)
      POST /auth/refresh  →  new token  →  retry original request
      If refresh fails  →  clearToken() + dispatch 'auth:logout'
                              AuthContext.logout() → user=null → redirect /login
Security Note
The access token never touches localStorage or sessionStorage. tokenStore.js is a module-scoped variable — invisible to browser devtools storage panels.

GlobalTimerWatcher (App.tsx)

A headless component that ticks every 1 s, reads focus_timer_state_v1 from localStorage, and fires browser Notification API calls when a phase expires. This is the legacy display mirror — the authoritative timer state lives on the server. The two systems coexist: localStorage is only a cross-page display convenience.

05 File-by-File Breakdown

Backend

src/index.js

Express app factory. Mounts all route groups. Sets CORS origin whitelist (hardcoded; update for new deployments). No middleware applied globally except json + cookie-parser. authMiddleware and apiLimiter applied per-route-group — intentional because /auth routes have their own per-endpoint limiters.

src/modules/auth/auth.services.js

Three exports: verifyGoogleToken, findOrCreateUser, generateTokens. verifyGoogleToken validates the Google ID token locally (no extra HTTP round-trip), checks iss claim and email_verified. findOrCreateUser does a find-or-insert, then always calls generateTokens. generateTokens mints 15-min access JWT + 7-day refresh JWT, stores refresh token in DB.

src/modules/auth/auth.controller.js

HTTP layer for auth. Key security decisions:

  • Refresh token set as httpOnly; secure; sameSite=none cookie (7-day). Never in response body.
  • refresh handler checks user.refreshToken === token before issuing. Mismatch → wipe DB token + clear cookie + 401. This is replay detection.
  • deleteAccount cascades through all foreign keys (PostgreSQL ON DELETE CASCADE).
  • Classroom OAuth token stored separately; retrieved/cleared by dedicated endpoints.

src/modules/auth/auth.middleware.js

Reads Authorization: Bearer <token>. Verifies JWT. Injects req.user = { userId, email }. Returns { code: 'TOKEN_EXPIRED' } on expiry — the frontend api.js uses this specific code to trigger silent refresh.

src/modules/courses/courses.services.js

computeStats(course, evals) is the grade engine. Iterates evaluations, accumulates weighted grade, computes requiredAvg (what student needs on remaining evals to hit target). Returned in every GET course response. No DB aggregation — pure JS computation.

src/modules/evals/evals.services.js

assertCourseOwnership called at top of every mutation — prevents IDOR. Validates eval fields (type whitelist, weightage 0–100, score ≤ maxScore). When score is updated, calls onEvalScoreUpdated hook (triggers achievement check).

src/modules/focus/session.service.js

Most security-sensitive backend file. endSession flow:

  1. If serverToken present: find session.start event, check not consumed, compute duration from event timestamp (client actualMinutes ignored). Mark consumed.
  2. Clamp all inputs server-side.
  3. Enforce daily cap (20 sessions/day).
  4. Compute integrityScore from visibility + interaction data.
  5. Compute XP (capped at 300/session).
  6. Upsert stats + streak.
  7. Insert session event + sub-events (comeback, linked, time-of-day).
  8. Run achievement evaluation.
  9. Award achievement XP idempotently.
  10. Return everything for client display.
Timer Path
When called from timer.service.js, serverToken is null and actualMinutes is already server-computed. Steps 1–2 (token anchor) are skipped.

src/modules/focus/timer.service.js

Server-owned timer. One row per user in timers table (unique constraint). Every operation is idempotent: start returns existing timer if one exists; pause/resume are no-ops if already in target state. nonce = 16-byte hex random, generated on start, echoed back on end — prevents replay of the same timerEnd request twice.

src/modules/focus/achievements.js

Achievement registry. Each entry has: id, emoji, tier, xp, hidden, name, desc, rule (typed evaluation spec), triggers (event types that can unlock this). Also exports ACHIEVEMENT_MAP (Map by id) and TRIGGER_INDEX (Map from event type → Set of achievement ids) for O(1) lookups.

src/modules/focus/utils/achievements.eval.js

evaluateAchievements(userId, ctx) — master evaluator. Flow: narrow candidates via TRIGGER_INDEX → evaluate each rule via switch statement → lazy-load DB data (each source at most once per run) → batch-insert newly qualified achievements with onConflictDoNothing → return only newly inserted rows.

src/db/gamification.js

All gamification queries. Key: upsertStats uses raw SQL INSERT … ON CONFLICT DO UPDATE that atomically increments counters and resets today/week minutes on date rollover. Level computed inline: GREATEST(1, FLOOR(SQRT(total_xp / 100)) + 1).

src/common/utils/xp.js

computeIntegrityScore scores a session 0–1 based on duration ratio, visibility ratio, and interaction count. computeXP derives raw XP (base + plan-bonus × multiplier, cap 300). awardAchievementXpIdempotent checks event log before awarding — idempotency guard for XP.

Frontend

src/lib/api.js

All API calls through request(method, path, body, _retry). On 401 TOKEN_EXPIRED and !_retry: fires POST /auth/refresh, stores new token, retries original. On any other non-2xx: throws Error with .status and .data attached. _retry flag prevents infinite loops.

src/lib/tokenStore.js

Three-line module. _token is a module-scoped variable. Intentionally not in React state — must be readable synchronously inside api.js's headers() function without React context.

src/lib/sessionCache.ts

In-memory Map<string, CacheEntry>. All keys centralised in CACHE_KEYS. cacheInvalidate(...keys) deletes specific keys. cacheClear() empties all (called on logout). No TTL — cache lives until explicit invalidation or tab close.

src/lib/dataService.ts

primeCache() fetches courses + upcoming evals in parallel, then fetches each course detail in parallel. All stored in sessionCache. Public functions check cache first. Invalidation functions wrap cacheInvalidate / cacheClear with domain logic.

src/context/AuthContext.tsx

Mount effect: fires refresh + me. Sets token in memory, user in state. Registers auth:logout DOM event listener. login() and logout() are stable callbacks via useCallback.

src/context/AchievementContext.tsx

Manages unlock queue. Key design decisions: seenRef persists via useRef seeded from localStorage; new achievements marked seen before enqueuing (prevents duplicates on rapid poll cycles); cinematic-tier achievements sorted to end of queue (toasts display first); fetchingRef prevents concurrent fetches; audio warmed on first user click (browser autoplay policy).

src/pages/FocusTimerPage.tsx

Most complex page. Two timer modes coexist: (1) Server-owned timer — fetched on mount, synced via POST /focus/timer/sync every 10 s, displayed via local countdown. (2) Break phase — purely client-side 5-min countdown after session ends. Integrity tracking: invisibleRef accumulates tab-hidden seconds; interactionCountRef counts throttled mouse/key events. Both sent to /focus/timer/end.

06 Function-Level Deep Dives

computeIntegrityScore({ plannedMinutes, actualMinutes, invisibleSeconds, interactionCount })

Scores a session 0–1 to modulate XP. Prevents gaming by running a session in a background tab or going AFK.

durationScore = ratio > 1.5 → 0.3 (significantly over-ran) ratio < 0.5 → 0.4 (gave up early) 0.85–1.15 → 1.0 (on-plan) else → 0.75 visibilityScore = invis > 50% → 0.1 (mostly hidden) invis > 30% → 0.5 invis > 10% → 0.8 else → 1.0 interactionScore= 0 events → 0.3 (probably AFK) < 10% expected → 0.5 < 30% expected → 0.75 else → 1.0 finalScore = duration × 0.4 + visibility × 0.35 + interaction × 0.25

computeXP({ actualMinutes, integrityScore, plannedMinutes })

base = min(actualMinutes × 2, 150) planBonus = |actual - planned| < 5 ? 10 : 0 mult = integrityScore < 0.4 → 0 integrityScore < 0.6 → 0.5 else → 1.0 xp = min((base + planBonus) × mult, 300)

upsertStats(userId, minutes, xp, today, weekStart, isNewSession)

Atomically updates user_stats row. The CASE WHEN stats_date = today branch resets today_minutes to just the new session if the date changed. Level is computed inline in SQL: GREATEST(1, FLOOR(SQRT(total_xp / 100)) + 1) — always consistent with total XP.

Two concurrent session endings for the same user can race here. PostgreSQL ON CONFLICT DO UPDATE is atomic per-row so no corruption, but achievement evaluation reads stats after — the second eval might see stats that include both sessions. Acceptable for current single-user design. Fix: wrap in FOR UPDATE transaction.

upsertStreak(userId, today, yesterday)

No streak row      → insert currentStreak = 1
lastActiveDate === today     → no-op (already counted)
lastActiveDate === yesterday → increment, update longest
else                         → reset to 1 (broken)

Returns { currentStreak, wasIncremented, wasBroken, previousStreak }. The previousStreak field qualifies phoenix/rebuilt achievements that require breaking a long streak and rebuilding.

evaluateAchievements(userId, ctx)

Input Description
stats user_stats row
streak user_streaks row
earnedSet Set<achievementId> already earned — prevents re-award
sessionMeta { actualMinutes, integrityScore, hourOfDay, daysSinceLastSession, hoursBeforeEval, todaySessionCount }
triggerEvents Array of event types to narrow candidates via TRIGGER_INDEX

Lazy loading: getCourses, getArchived, getSessions each load at most once via ??=. Avoids N×DB calls when many achievements need the same data. Idempotency: insertAchievements uses onConflictDoNothing. Returns only newly inserted rows.

startTimer / endTimer

Operation Key behaviour
start plannedMinutes clamped [1, 240]. nonce = 16-byte hex random. If timer exists, return it (idempotent — handles network retries).
end Verify body.nonce === timer.nonce (replay protection). Compute elapsed from DB timestamps. Delete timer row immediately (prevents duplicate-end race). If elapsed < 1 min or abort → dropped: true, no XP.

primeCache() — dataService.ts

Called on first cache miss of any key. Fires getCourses() + getUpcomingEvals() in parallel, then getCourse(id) for each course in parallel. Stores all results. The first page pays the full cost; all subsequent pages are instant. No TTL — data stays fresh until a mutation explicitly invalidates it.

  ▼

User clicks "Sign in with Google"
  │
  ▼
Google Identity Services SDK fires callback with idToken
  │
  ▼
POST /auth/google
  ├─ verifyGoogleToken(idToken)          validates with Google's public keys; checks iss + email_verified

  ├─ generateTokens(user)
  │   ├─ JWT (15m) access token
  │   └─ JWT (7d) refresh token  →  stored in DB + set as httpOnly cookie

  │
  ▼

Refresh Token Rotation

API call returns 401 TOKEN_EXPIRED
  │
  ▼
api.js: POST /auth/refresh (cookie, _retry not set)
  │
  ├─ Server: verify cookie JWT signature
  │           match cookie value vs DB user.refreshToken
  │           MISMATCH  →  wipe DB + clear cookie + 401  (replay detected)
  │
  ├─ 200  →  setToken(new access token)
  │         retry original request with _retry=true
  │
  └─ Failure  →  clearToken() + dispatch 'auth:logout'
               AuthContext.logout()  →  user=null  →  redirect /login

Focus Session (Full Timer Flow)

User configures timer (duration, optional task/eval link)
  │
  ▼
POST /focus/timer/start { plannedMinutes, linkedTaskId?, ... }
  ├─ server generates nonce, stores timer row (startedAt = now())
  └─ returns { timer: { nonce, remainingSeconds, ... } }
  │
  ▼
Client: starts local countdown from remainingSeconds
        polls POST /focus/timer/sync every 10s
        tracks invisibleSeconds (visibilitychange) + interactionCount (mouse/key)
  │
  ▼  (timer hits 0  OR  user clicks end)
  │
POST /focus/timer/end { nonce, invisibleSeconds, interactionCount }
  ├─ nonce check  →  mismatch  →  dropped: true
  ├─ computeElapsedSeconds from DB timestamps
  ├─ deleteTimer  (race protection: duplicate end ignored)
  ├─ endSession(userId, { actualMinutes: serverComputed, ... })
  │   ├─ daily cap check (20 sessions/day)
  │   ├─ computeIntegrityScore + computeXP
  │   ├─ upsertStats + upsertStreak
  │   ├─ insertEvent session.completed + sub-events
  │   ├─ evaluateAchievements
  │   └─ awardAchievementXpIdempotent
  └─ returns { xp, integrityScore, newAchievements, stats, streak, goals }
  │
  ▼
Client: shows SessionComplete modal
        queues newAchievements for display
        AchievementContext.refresh() re-polls dashboard

Daily Goals Generation

GET /focus/dashboard
  │
  ├─ getTodayGoals(userId, today)  →  rows exist  →  return them
  │
  └─ EMPTY  →  generateDailyGoals(userId, stats, streak, today)
      ├─ get7DayAvgMinutes
      ├─ getGoalCompletionRate (last 1 and 3 days)
      ├─ compute baseline minutes
      │   isNewUser (<5 sessions)  →  15 min
      │   bad yesterday            →  60% of 7-day avg
      │   on completion streak     →  add stretch goal
      ├─ up to 3 goals:
      │   1. session_duration (always)
      │   2. eval_prep (eval ≤3 days away)  or  streak_maintain
      │   3. stretch (if completion streak)
      └─ insertGoals  →  return

Achievement Evaluation (Targeted)

Event fires (e.g. session.completed)
  │
triggerEvents = ['session.completed', 'session.linked_completed', ...]
  │
evaluateAchievements(userId, { ..., triggerEvents })
  │
  ├─ TRIGGER_INDEX.get('session.completed')  →  Set of achievement IDs
  │   union with other trigger IDs
  ├─ filter out already-earned
  ├─ for each candidate:
  │   switch(rule.type)
  │     'stat'         →  stats.totalSessions >= rule.gte
  │     'streak'       →  streak.currentStreak >= rule.gte
  │     'event_count'  →  countEventsByType(userId, rule.eventType)
  │     'session_meta' →  check sessionMeta fields
  │     ...
  ├─ batch insertAchievements (onConflictDoNothing)
  └─ return newly inserted rows

08 Database Design

Schema

users
  id (uuid PK), googleId (unique), email (unique)
  refreshToken            // null = logged out
  googleAccessToken / googleTokenExpiry  // classroom OAuth

courses
  id (uuid PK)
  userId → users.id CASCADE DELETE
  name, credits, targetGrade, isArchived

evaluations
  id (uuid PK)
  courseId → courses.id CASCADE DELETE
  title, type, date, weightage, maxScore
  score (nullable)        // null = not yet graded

events
  id (uuid PK)
  userId → users.id CASCADE DELETE
  type (text)             // "session.completed", "course.created", "achievement.xp.first_light", ...
  occurredAt, metadata (jsonb)
  INDEXES: (userId, type), (userId, occurredAt)

user_stats              // one row per user (PK = userId)
  totalSessions, totalMinutes, todayMinutes, weekMinutes, totalXp, level
  statsDate             // to detect day rollover in upsert
  weekStart             // for week rollover

user_streaks            // one row per user (PK = userId)
  currentStreak, longestStreak, lastActiveDate, freezeCount

user_achievements
  (userId, achievementId) UNIQUE INDEX  // prevents duplicate awards
  tier, xpAwarded, earnedAt

daily_goals
  userId, date, type, title, targetValue, status, xpReward
  INDEX: (userId, date)

timers
  userId UNIQUE          // one active timer per user — DB-level enforcement
  status ('running' | 'paused')
  startedAt, plannedMinutes
  pausedAt, pausedElapsedSeconds
  nonce                  // anti-replay token
  linkedTaskId, linkedEvalId, linkedEvalDueDate, quickTitle, quickCategory

Key Design Decisions

Decision Reason
events as ledger Rather than maintaining counters that drift, gamification data is derived by counting event rows. History queryable, achievements re-evaluable, idempotency trivial.
timers.userId UNIQUE Enforces one active timer per user at DB level, not just application level. Concurrent start requests — only one insert succeeds.
user_achievements unique index on (userId, achievementId) onConflictDoNothing makes the evaluator callable multiple times safely. Business logic can never double-award.
Cascade deletes everywhere Deleting a user removes all their data atomically. No orphaned rows.
refreshToken = null = logged out No blacklist table needed. Refresh endpoint checks DB value against cookie — simple, correct.
Level in SQL GREATEST(1, FLOOR(SQRT(total_xp / 100)) + 1) computed inline in upsert — always consistent with XP, no separate update needed.

10 Security Architecture

Authentication Flow

Google OAuth one-tap/button flow. Client receives Google idToken, forwards to POST /auth/google. Server verifies token signature against Google's public keys (no extra HTTP round-trip), checks iss and email_verified. Client never has direct access to Google's identity infrastructure.

JWT Tokens

Token Expiry Payload Storage
Access 15 min { userId, email, issuer: "semTracker" } Memory only (tokenStore.js)
Refresh 7 days { userId, issuer: "semTracker" } httpOnly cookie + DB

Refresh Token Rotation + Replay Detection

On every refresh, a new refresh token is issued and stored in DB. Old token becomes invalid. If user.refreshToken !== cookie, the existing DB token is wiped and the request rejected — detects token theft/replay. An attacker who steals and uses a refresh token will trigger this on the victim's next refresh, invalidating the attacker's session.

Rate Limiting

Endpoint Window Max Why
POST /auth/google 15 min 10 Brute force / scraping
POST /auth/refresh 15 min 30 Token refresh abuse
POST /auth/logout 5 min 10 Session invalidation spam
All /courses, /evals, /focus 1 min 100 General API protection
POST /focus/session/start 1 min 10 Session farming
POST /focus/session/end 1 min 25 XP farming — tighter than api limiter
POST /focus/track/task 1 min 30 task.completed event flooding
POST /focus/track/page 1 min 20 Event flooding
POST /focus/timer/start 1 min 5 One timer per user anyway (DB enforces)
POST /focus/timer/end 1 min 10 Nonce check already stops replays; secondary defense

Authorization

Every resource mutation asserts ownership. The pattern in evals:

const existing = await findEvalById(evalId);
if (!existing) throw ApiError.notFound(...)
if (existing.course.userId !== userId) throw ApiError.forbidden(...)  // IDOR prevention

Timer Anti-Cheat

  • startedAt set by server — client cannot forge start time
  • plannedMinutes clamped to [1, 240] server-side
  • nonce prevents replay of /timer/end
  • actualMinutes always computed from DB timestamps, never from client
  • interactionCount + invisibleSeconds accepted but clamped to physically possible bounds

Event Injection Prevention

const VALID_PAGES        = new Set(["progress", "settings"]);
const VALID_TASK_ACTIONS = new Set(["created", "completed"]);

Client cannot inject arbitrary event types via track endpoints. Anything outside these allowlists silently returns { newAchievements: [] }.

11 Gamification Engine

Session End Pipeline

Session ends (timer or legacy)
  │
  ├─ computeIntegrityScore (visibility + interaction + duration)
  ├─ computeXP (base + plan bonus) × multiplier, capped 300
  ├─ upsertStats (total/today/week minutes, XP, level)
  ├─ upsertStreak (today vs yesterday IST date comparison)
  ├─ insertEvent session.completed + sub-events
  ├─ generateDailyGoals (if first session of day and no goals exist yet)
  ├─ checkAndCompleteGoals
  └─ evaluateAchievements → insertAchievements (onConflictDoNothing)
      └─ awardAchievementXpIdempotent

Level Formula

level = max(1, floor(sqrt(totalXp / 100)) + 1) // computed in SQL, not application code Level 2 → 100 XP Level 3 → 400 XP Level 5 → 1600 XP // quadratic curve

Achievement Tiers

Tier Display XP range
bronze Toast notification 25–75
silver Toast notification 100–250
gold Toast notification 300–500
platinum Cinematic unlock 500–1000
legendary Cinematic unlock —
hidden Cinematic unlock (only revealed on earn) —

Hidden achievements have their id, name, desc, emoji, tier all stripped by buildCatalog() / sanitizeAchievements() until earned. Client sees { hidden: true, locked: true, earned: false } only.

Daily Goals Logic

Generated per-user per-day on first dashboard fetch after midnight (IST). Adapts to recent performance:

  • New user (<5 sessions): 15-min baseline, max 2 goals
  • Had a bad yesterday: reduce baseline to 60% of 7-day average
  • Completion rate 100%: add stretch goal
  • Upcoming eval within 3 days: replace streak goal with eval prep goal

Streak Freeze

Users have freezeCount (default 2). POST /focus/streak/freeze decrements it and inserts a streak.freeze_used event. Freeze does not modify lastActiveDate — it's a UI feature only. Streak calculation is unchanged.

12 Timer System

Two Coexisting Timer Modes

Mode State location Purpose
Server-owned timer timers DB table + serverTimer React state Authoritative for all duration/XP calculations. Survives page refreshes.
Legacy client-side timer focus_timer_state_v1 in localStorage Cross-page display mirror. GlobalTimerWatcher reads it to fire browser Notification API.

Timer State Machine

[no row in timers table]
        │
        ▼  POST /focus/timer/start
   [status: 'running']
        │
        ├─ POST /focus/timer/pause  →  [status: 'paused']
        │                                    │
        │  POST /focus/timer/resume ←────────┘
        │
        ├─ POST /focus/timer/extend  →  plannedMinutes += addMinutes (capped 240)
        │
        ├─ POST /focus/timer/sync   →  returns current state (10s heartbeat)
        │
        └─ POST /focus/timer/end or /abort  →  [row deleted]  →  endSession or dropped

Elapsed Time Computation

computeElapsedSeconds(timer):
  baseSeconds = floor((now - startedAt) / 1000)
  paused      = pausedElapsedSeconds
  if status === 'paused' && pausedAt:
    currentPauseSec = floor((now - pausedAt) / 1000)
    return max(0, baseSeconds - paused - currentPauseSec)
  return max(0, baseSeconds - paused)

Client-supplied elapsed is never used for XP. Server recomputes from startedAt (immutable, set on insert) minus accumulated pausedElapsedSeconds.

13 Scalability & Bottlenecks

Current Limits

In-memory rate limiting: express-rate-limit stores counters in process memory. On multiple server instances, rate limits are per-instance, not global. Fix: rate-limit-redis with a shared Redis instance.
IST hardcoded: dates.js uses Asia/Kolkata everywhere. International users have streaks/goals in IST, not their local time.
No DB pool config: db/index.js uses Drizzle's default pool. Under heavy concurrent load, connections could exhaust. Add max: N to pool config.
Achievement evaluation is the hot path: evaluateAchievements fires 10–20 queries per session end. Under load, this is the dominant cost of POST /focus/timer/end.

What's Already Optimized

Optimization Where Impact
TRIGGER_INDEX achievements.js Narrows evaluation candidates ~60–80%, skipping achievements that can't fire on the current event type.
Lazy loaders in evaluator achievements.eval.js Each DB source (courses, sessions, etc.) loaded at most once per evaluation run via ??=.
onConflictDoNothing gamification.js Duplicate achievement evaluation runs are safe and cheap — DB handles deduplication.
primeCache() dataService.ts All course data loaded in one burst on first miss. Subsequent pages are instant.

Future Optimizations

Optimization What it fixes
Redis for rate limiting Required before horizontal scaling
Achievement eval as background job (BullMQ) Move out of request path; session end returns immediately, achievements appear seconds later via polling
Composite index on (userId, type, occurredAt DESC) Events table as it grows with many users × many sessions
Single getCourses?withEvals=true endpoint Replace N parallel getCourse calls in primeCache()
React.lazy for heavy pages FocusTimerPage, ProgressPage — reduce initial bundle

14 Developer Modification Guide

Adding a New Backend Route

  1. Create src/modules/<domain>/<domain>.routes.js, .controller.js, .services.js
  2. Import and mount in src/index.js with authMiddleware + apiLimiter
  3. Add a dedicated rate limiter in rateLimiter.js if endpoint could be abused
  4. Add query functions to db/queries.js or db/gamification.js

Adding a New Achievement

  1. Open src/modules/focus/achievements.js
  2. Add entry to ACHIEVEMENTS array with id, emoji, tier, xp, hidden, name, desc, rule, triggers
  3. If new rule.type needed: add a case in achievements.eval.js
  4. Add trigger event to triggers array — TRIGGER_INDEX builds automatically
  5. If achievement needs a hook (fires on course creation etc.): call evaluateAchievements in relevant hooks.service.js function

Adding a New Page (Frontend)

  1. Create src/pages/NewPage.tsx
  2. Add route in App.tsx (wrap in <ProtectedRoute> if auth-required)
  3. Fetch data via dataService.ts (reads) or api.js (mutations)
  4. If page-visit achievement wanted: add to VALID_PAGES in focus.controller.js, then call trackPageVisit('new_page')

Modifying XP/Level Formula

Changing the level formula does not retroactively recalculate existing user levels. Run a migration: UPDATE user_stats SET level = GREATEST(1, FLOOR(SQRT(total_xp / 100.0))::int + 1);
What to change Where
XP per session computeXP in src/common/utils/xp.js
Level from XP Inline SQL in upsertStats in src/db/gamification.js
Integrity score weights computeIntegrityScore in xp.js

Avoiding Cache Bugs (Frontend)

After this mutation… Call this
Create / delete / archive course invalidateAllCourseData()
Create / update / delete eval invalidateCourseDetail(courseId)
Any achievement / gamification data Nothing — not in sessionCache
Logout invalidateOnLogout() (called by AuthContext.logout)

Never read cacheGet without checking cacheHas first — returns undefined on miss, easily confused with a stored undefined.

Avoiding Achievement Bugs

  • Never call insertAchievements directly for business logic — always go through evaluateAchievements. The evaluator handles earned-set filtering and batch insert.
  • awardAchievementXpIdempotent checks countEventsByType(userId, 'achievement.xp.<id>') before awarding. This is the XP idempotency guard. Don't bypass it.
  • Achievement triggers must be added to the triggers array in the definition, not just as switch cases. TRIGGER_INDEX is built from triggers.
